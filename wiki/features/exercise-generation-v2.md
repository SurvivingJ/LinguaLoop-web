## 1. Discovery Summary

### What the product is

LinguaDojo is a server-rendered Flask + Supabase web app for **active, scored language practice** — comprehension tests (reading/listening/dictation/pinyin/pitch-accent), a per-word **Vocabulary Ladder** (Nation-grounded, 9 levels in 4 rings across 6 cognitive families), FSRS flashcards, and a merged **Practice Engine** that ranks every candidate item by a unified score (`α·ladder + β·IRT + γ·BKT + δ·FSRS`). All content is system-generated by LLM pipelines run from a local admin dashboard. Target languages: **Chinese (1), English (2), Japanese (3)** — all `is_active`.

**The platform is pre-launch**: 11 users, **0** rows in `user_word_ladder`, `user_vocabulary_knowledge`, `user_flashcards`, `exercise_attempts`. There is no user data to migrate and no live behaviour to preserve — breaking schema changes are nearly free *now* and expensive after launch. Phase 0 deliberately front-loads every schema decision (script variants, register, semantic_class enum, content schema) for exactly this reason.

### The two existing pipelines, compared (and the consolidation verdict)

| Dimension | `services/exercise_generation` (legacy) | `services/vocabulary_ladder` |
|---|---|---|
| Unit of work | **Sentence-first**: builds a sentence pool (transcript mining via `TranscriptMiner` + LLM top-up), then stamps out N exercises per type from the pool (16/sense `VOCABULARY_DISTRIBUTION`) | **Sense-first**: P1 builds a canonical per-sense ground-truth asset (definition, POS, primary collocate, 10 tier-pinned sentences, morph forms); P2/P3 derive level exercises *from the asset* |
| Output labels | `exercise_type` only — no `ladder_level`, family, or variant | `ladder_level` + family + A/B variant + `word_asset` provenance — what `get_practice_session` keys on |
| QA | Cloze judge + (2026-06-10) sentence-validity on semantic; flashcards/translations unjudged | Full 4-judge layer incl. the P1 sentence judge that gates everything downstream; per-language validation profiles |
| Reusability | Exercises only | `word_assets` ground truth reusable for **new** exercise types without re-deriving word facts |
| Sources | vocabulary + grammar/collocation/conversation/style | vocabulary senses only |

**Decision (operator-confirmed): the vocabulary ladder is the sole source of exercise generation for senses.** The sentence-first/sense-first distinction is historical, not architectural. The legacy pipeline's one unique asset — **transcript mining** (sentences from tests learners actually encounter; good contextual encoding, zero LLM cost) — is folded into the ladder as a *P1 sentence source* (P1.2): mined candidates are tier-filtered and pass the P1 sentence judge alongside generated ones. The legacy pipeline is then **frozen**: it remains the dormant home for grammar/conversation/style sources (all outside Practice V1 per ADR-012; the 7,235 sense-less conversation exercises are explicitly deferred per operator answer), receives no further investment, and its `VOCABULARY_DISTRIBUTION` path is removed.

### Standalone trainers (reused, not duplicated)

**Pinyin tone trainer** (ZH passage-level tone ID with sandhi), **pitch-accent trainer** (JA), **classifier drill** (ZH 量词: 75 classifiers / ~2.6k pairs / 16 distractor groups, deterministic serving, LLM curation pipeline), **furigana overlay**, **dictation**. The classifier drill becomes the ZH L4 sub-module (§5 #6); its architecture is the confirmed template for a JA counter drill (§7 P3.4). The deterministic preprocessors behind the trainers (pypinyin/jieba, pyopenjtalk, fugashi/UniDic) are the enrichment tools for Phase 0 backfills.

Serving, scheduling, mastery, and difficulty calibration are **already solved and live in SQL**: `get_practice_session` + `dim_practice_modes` + `dim_exercise_types` exist in the live DB (the practice-engine wiki page's `planned` status is stale), BKT per sense with decay/inference, FSRS-4.5, dual ELO, nightly IRT 2PL. **This plan designs the content factory that feeds the existing machine — not a new serving algorithm.**

## 2. Schema Analysis

### Vocabulary & content tables (live state, 2026-06-11)

**`dim_vocabulary`** (7,461: ZH 3,890 / EN 3,571 / JA 0) — `id, language_id, lemma, phrase_type, component_lemmas[], part_of_speech (98%), frequency_rank (real, ~90%), level_tag, semantic_class (≈0%)`. **No script-variant column** (Traditional Chinese groundwork added in P0.9), no reading/tone fields (readings live on senses, sparsely).

**`dim_word_senses`** (17,556: ZH 8,084 / EN 9,472 / JA 0) — `vocab_id, definition_language_id, definition, pronunciation (≈0%), ipa_pronunciation (≈0%), example_sentence, usage_notes, sense_rank, usage_frequency, semantic_category (≈0%), morphological_forms jsonb (≈0%), definition_level, is_validated, gen_confidence`. Per-sense tracking is canonical (ADR-002). `definition_language_id` already exists — the hook multi-nl support will hang from.

**`exercises`** (11,256) — JSONB `content`, `exercise_source_type` enum with exactly-one-FK check, `ladder_level`, `tags` (judge verdicts, variant), IRT columns, `is_active`. RLS disabled (content table, deliberate). ZH: 2,266 vocab + 117 conversation; EN: 1,620 vocab + 7,235 conversation; ladder-linked: 80.

**`word_assets`** — per-sense ground truth (P1/P2/P3, A/B variants), `is_valid`, `validation_errors`, `validation_warnings[]`, `prompt_version`. 12 senses covered.

**User-state tables** (all RLS-enabled, all empty): `user_vocabulary_knowledge`, `user_flashcards`, `user_word_ladder` (incl. `active_levels[]`, `family_confidence`), `user_exercise_history`. `users.exercise_preferences` JSONB exists — the natural home for the script-variant toggle.

**Serving objects (live)**: `get_practice_session`, `get_exercise_session` (wrapper), `get_ladder_session`, `dim_practice_modes`, `dim_exercise_types` (13 rows, `expected_seconds`=45 flat, family mis-map G4).

**`prompt_templates`** (model-assignment source of truth, unique `(task_name, language_id, version)`): healthy for EN/ZH generation + judges post-fix; JA rows exist only for tl_nl/nl_tl; JA needs sentence-gen, P1/P2/P3, 4 judges, cloze activation (P0.8).

**CJK payloads** live at the *test* level (`pinyin_payload`, `pitch_payload`, furigana) — built by deterministic preprocessors (pypinyin/jieba, pyopenjtalk, fugashi/UniDic) that double as Phase-0 enrichment tools for per-sense readings.

**Classifier dictionary** (ZH): `dim_classifiers` + pair tables + offline LLM curation pipeline (`services/classifier_curation/`). Ready-made data source for ladder L4-ZH and the architectural template for JA counters.

### Constraints visible from the schema

- **Scale is tiny** (single-digit MB). Pre-generation storage is a non-issue; pgvector installed (topic dedup today, sense embeddings in P2.2).
- **RLS**: content tables anon-readable by design; generation uses the service-role client from the local admin app. No obstacle.
- **Batch infrastructure**: admin daemon threads + SSE + `is_task_stopped()` is the established runner; APScheduler with advisory locks for nightly jobs. No external queue exists; none needed at this scale.

## 3. Intent Map (operator-ratified)

1. **Core goal.** A single content factory — the ladder pipeline — that fills the existing adaptive serving machine: per-sense, judge-gated, generate-once exercises across the cognitive families, with enough coverage that `get_practice_session` always finds an item for the family it targets, in **all three languages from the start (JA now)**.
2. **Triggers.** Batch/operator-driven (admin tabs, pack creation) + async top-up via `generation_queue` on sense subscription. Never serve-time generation — "generate-once, use-forever" (assessment validity, IRT stability) is load-bearing.
3. **Quality bar.** Assessment-grade: stable items, exactly one defensible key, every LLM-authored item judged, judges *act* (block/replace) and **fail closed in batch**.
4. **Language coverage.** EN/ZH/JA parity in the first batch (top 1,000 senses each). The capability matrix (§6.2) makes per-language behaviour a data row, not a code fork. **Traditional Chinese is in scope** as a user-switchable script variant (§6.7).
5. **Native language.** **nl = English for V1, multi-nl is coming.** Design rule now: TL-facing content is nl-free (already the numeric-key TL-immersion convention); every nl-facing string is stored under an nl-keyed map so adding a language appends keys instead of reshaping content (§6.4). No `'en'` literals in generation code paths.
6. **AI involvement.** Hybrid: LLM for sentence/distractor/reasoning authoring (DB-driven per-language models), LLM-as-judge for QA, **deterministic wherever a dictionary or algorithm suffices** (definition match, jumbled, readings, tone, classifier/counter). No runtime LLM (L10 capstone is **off** per operator decision; `cloze_typed` grades by normalised match).
7. **User control.** Packs, time budgets, and now a **script-variant toggle** (simplified/traditional). Exercise-type preferences remain Phase 4.

## 4. Pedagogical Framework (applied to this project)

The existing design is unusually well-grounded; this section names what to preserve and where current generation violates it.

**Retrieval practice & desirable difficulties — preserved by serving, violated by content gaps.** The ladder's receptive→productive ring progression, first-attempt-only credit, cross-session gating, and A/B variants are textbook retrieval practice. The violation is inventory: when a required family has no exercise for a sense, difficulty becomes noise. **The pipeline's contract: per sense, ≥1 active exercise for every family its `active_levels` require** — enforced by the end-of-batch coverage check (§6.3).

**Spaced repetition — extend, never duplicate.** FSRS owns post-mastery; momentum bands own pre-mastery; BKT decay flags re-entry. The pipeline's SRS obligation is **variant supply** (≥2 per type per sense) so repeat retrievals aren't item memorisation. Already in the ladder pipeline; the legacy "same 3 sentences across 4 types" anti-pattern dies with consolidation.

**Contextual encoding.** P1's judged 10-sentence base is the backbone every item inherits. Two rules: sentences pinned to the sense's tier (the eval's "C2 sentence for an A1 word" failure → hard gate in P2.5), and mined transcript sentences (learner-encountered contexts) preferred when they pass the same judge (P1.2).

**Nation's framework → the 6 families.** form_recognition (L1/L2) + meaning_recall (L3) = meaning-focused input; form_production (L4/L9), collocation (L5/L8), semantic_discrimination (L6/L7) = language-focused learning; `timed_speed_round` = fluency development. **contextual_use stays unbuilt** (operator: leave off) — consequence: family weight 0.18 caps overall p_known at ≈0.92 and stress-test composition falls back to the highest available level for its 2 contextual_use slots (existing behaviour, acceptable for now; revisit post-launch).

**Language-specific handling:**

- **Chinese.** Character recognition, pinyin reading, and tone knowledge are separable skills — add per-sense `hanzi_to_pinyin`, `tone_id_word` (deterministic; Phase 1) alongside the passage-level pinyin trainer. L4-ZH = `classifier_match` from the classifier dictionary (never 个; semantic-group distractors). L1 distractors stay **audio-confusable only** (settled policy; judge-enforced); visual-similarity distractors belong to `pinyin_to_hanzi` (Phase 3). **Traditional script**: canonical storage stays simplified; serve-time conversion + override table + per-user toggle (§6.7).
- **Japanese.** `kanji_to_reading`/`reading_to_kanji` keyed to the **P1-sentence contextual reading** (on/kun disambiguation via UniDic), `particle_selection` as L4-JA (the highest-value JA addition; は/が, に/で, を/が confusion drives distractors), morphology_slot for verb/adjective conjugation. Pitch accent stays with the dedicated trainer (no per-sense duplication in V1). **Counters (助数詞): build as a clone of the classifier-drill architecture** (curated dictionary + LLM curation pipeline + deterministic serving) — operator-confirmed approach (P3.4). **Keigo: track register per JA sense** (operator answer 13) — new `dim_word_senses.register` column populated by the JA P1 prompt (`plain | polite | honorific | humble | formal | casual`), consumed by L6 register-error sentences, flashcard usage notes, and (later) production grading.
- **English.** Morphology slot and jumbled sentence fit as-is. Additions: `word_family` (derivational morphology from enriched `morphological_forms`) and corpus-grounded collocations (P2.4 fixes G6). Stress/homophone audio items: **excluded** (operator answer 12).

**Parts of speech.** `active_levels` (computed from `semantic_class` at ladder-row creation) is the routing primitive. Blocker G3: `semantic_class` is empty, so every word currently gets all 9 levels — the eval's "bean" failure. **The controlled vocabulary is ratified before backfill (P0.2)**, per operator answer 7:

| `semantic_class` | Covers | Routing consequences |
|---|---|---|
| `concrete` | physical objects, animates | skip L5/L8 (no tight collocates); L4-ZH = classifier, L4-JA = counter+particle, L4-EN = plural morphology |
| `abstract` | abstract nouns, concepts | full ladder incl. L5/L8 |
| `action` | verbs, processes | full ladder; L4 = conjugation (EN/JA) / aspect particle (ZH) |
| `property` | adjectives, adverbs | full ladder; L4 = comparison/conjugation where the language inflects |
| `function` | particles, conjunctions, pronouns, measure words themselves | L1–L3 + L6/L7 only; no collocation, no morphology slot |
| `proper` | proper nouns, acronyms | **excluded from ladder subscription**; definition-flashcard only |

Stored as a CHECK constraint on `dim_vocabulary.semantic_class`; `LANGUAGE_VALIDATION_PROFILES` and the capability matrix key on these six values.

**Distractor doctrine, per mechanism:**

- **Semantic** (cloze, definition match, syn/ant): same POS, same `definition_level` band, frequency_rank within ~±1 band, embedding cosine in a mid window (P2.2 makes this a SQL filter). Never morphological variants of the key.
- **Phonological** (L1, readings, tone): deterministic confusion sets — ZH: same syllable different tone > near initial/final > homophone-different-character; JA: long/short vowel, voiced/unvoiced, っ, mora count; real words only (judge-enforced).
- **Visual/orthographic** (reading→character): shared radical/component (ZH), confusable kanji (JA) — needs the small character-component table (P3.3).
- **Grammatical-paradigm** (morphology, particles, classifiers, counters): drawn from the closed paradigm itself; the judge checks key *uniqueness*, not plausibility.
- **Universal invariants**: exactly one defensible key (judged); options register-matched; `distractor_tags` carry the *specific* error category (the generic `"semantic"` tag is a defect).

## 6. Pipeline Architecture

### 6.1 Consolidation (decided)

`services/vocabulary_ladder` is the **only** generator of sense-linked exercises. Concretely:

- `VOCABULARY_DISTRIBUTION` and the vocabulary branch are removed from the legacy orchestrator; the admin "Exercise Generation → vocabulary" action routes to the ladder pipeline (P1.1).
- Transcript mining moves into P1 as a sentence *source*: mined sentences are tier-filtered, judged by the P1 sentence judge, and mixed with generated sentences (target ratio configurable; mined preferred when they pass) (P1.2).
- Existing legacy vocab exercises stay served per sense until ladder coverage supersedes that sense, then are deactivated (`is_active=false`, never deleted — IRT/attempt history) (P1.7).
- The legacy pipeline is frozen as the dormant grammar/conversation/style engine — no investment, no removal (sense-less content is a later issue by operator decision).

### 6.2 Capability matrix — the routing core

One table replaces scattered config (audit B3.3; the 小熊 incident is the proof):

```sql
CREATE TABLE dim_exercise_capabilities (
    language_id    smallint NOT NULL REFERENCES dim_languages(id),
    type_code      text     NOT NULL REFERENCES dim_exercise_types(type_code),
    pos_classes    text[]   NOT NULL DEFAULT '{all}',   -- semantic_class values from §4's ratified enum
    ladder_level   smallint,                            -- NULL = non-ladder (timed_speed_round)
    generator      text     NOT NULL,                   -- 'deterministic' | 'llm' | 'hybrid'
    requires       text[]   NOT NULL DEFAULT '{}',      -- {'pronunciation','morph_forms>=2','classifier_dict','counter_dict','tts',...}
    judge_key      text,                                -- NULL only for deterministic generators
    is_enabled     boolean  NOT NULL DEFAULT true,
    PRIMARY KEY (language_id, type_code)
);
```

Seeds encode §5's Lang column — e.g. `(3,'particle_selection','{action,concrete,abstract}',4,'llm','{p1_sentences,tokenised_particles}','particle_judge',true)`; `(1,'morphology_slot',…,is_enabled=false)`. `compute_active_levels` and the generators read this table; adding a language or type is an INSERT. An invariant test asserts every `(language, semantic_class)` pair yields ≥1 enabled type per required family.

### 6.3 Data flow

```
TRIGGERS
  a) Admin batch ("generate top-N senses"; daemon thread + SSE)
  b) Pack creation (pack senses without assets enqueued)
  c) Top-up (sense subscribed without assets → generation_queue; async;
     ladder serves whatever families exist meanwhile)
        │
        ▼
ENRICHMENT GATE (deterministic, idempotent, per sense)
  pronunciation: pypinyin(+sandhi ctx) | fugashi/UniDic | EN: skip
  semantic_class: present? else LLM-classify (ratified 6-value enum)
  register (JA): from P1 output
  traditional lemma (ZH): OpenCC + override table          → writes back to dim tables
        │
        ▼
ROUTING  dim_exercise_capabilities × (language, semantic_class) → active_levels + generation list
        │
        ├─ deterministic generators (definition_match, jumbled, readings, tone,
        │   classifier/counter, speed-round composition)      — no LLM, no judge
        └─ LLM generators
             P1 (mined + generated sentences → sentence judge → ground truth)
             → P2/P3 + new types → per-type judges
             every LLM-authored item passes its judge_key or is BLOCKED;
             verdicts → exercises.tags.<judge>_judge + llm_calls(judge_verdict, cost_usd)
        │
        ▼
VALIDATION (JSON-schema gate per (type, schema_version)) → exercises rows
  (sense_id, word_asset_id, ladder_level, exercise_type, content, tags{variant,judges})
        │
        ▼
COVERAGE CHECK (end of batch): every required family per sense has ≥1 active item
  → shortfalls → generation_queue(reason='coverage_gap'); batch report via SSE
```

Serving is untouched; the pipeline's deliverables to `get_practice_session` are correct family mappings (P0.3) and inventory.

### 6.5 Pre-generation vs on-demand

| Class | Strategy | Why |
|---|---|---|
| All MC/ordering/reading/typed items | **Pre-generate** | generate-once ADR; 60–95 s/sense is far too slow for serve-time; storage trivial |
| TTS audio | Pre-generate, deterministic R2 slugs | idempotent re-render solved |
| `timed_speed_round` batteries | **Serve-time composition** | pure selection |
| Traditional-script mirror (ZH) | **Pre-generate** (`content.hant`, §6.7) | dual-store decided; serving does zero conversion |
| Top-up for new subscriptions | Async via `generation_queue` | don't block session start on a 90-second pipeline |

```sql
CREATE TABLE generation_queue (
    id            bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    sense_id      integer NOT NULL REFERENCES dim_word_senses(id),
    language_id   smallint NOT NULL,
    reason        text NOT NULL,                       -- 'pack'|'subscribe_topup'|'coverage_gap'|'regen'
    status        text NOT NULL DEFAULT 'pending',     -- pending|running|done|failed
    detail        jsonb,
    requested_at  timestamptz NOT NULL DEFAULT now(),
    completed_at  timestamptz,
    UNIQUE (sense_id, reason)
);
```

### 6.6 LLM usage strategy

- **Models stay DB-driven** in `prompt_templates`. Current healthy assignments: EN generation `google/gemini-3.5-flash`, EN judges `gemini-2.5-flash-lite`, EN ladder P2/P3 `anthropic/claude-opus-4-7`; ZH/JA generation `qwen/qwen3.7-plus`, ZH judge `deepseek/deepseek-chat`. New JA prompts start on `qwen/qwen3.7-plus` (established JA decision) with `anthropic/claude-sonnet-4-6` as the Model-Arena challenger before scale-up.
- **Slug-rot defence (mandatory):** nightly `model_slug_health` job — `SELECT DISTINCT model FROM prompt_templates WHERE is_active` probed against OpenRouter `/models`; misses flip a dashboard banner + ERROR log. Judges **fail closed in generation batches** (block + alert); fail-open only at serve-adjacent call sites.
- **Caching:** generated assets are the cache. Enrichment results persist to dim tables (no repeated NLP/LLM calls per sense). Static system prefixes per task for provider prompt caching.
- **Quality gates, in order:** JSON-schema validation → deterministic checks (whole-word target presence per language tokenizer, option counts, key-duplicate scan) → judge (Likert-5 per `distractor-judge-v3-likert`) → persist with verdict. Every judge writes `llm_calls.judge_verdict/judge_confidence/cost_usd` (finish the 2026-06-10 `log_judge_verdict` rollout across all judges).

**New prompt drafts** (seed `prompt_templates`, numeric keys per house style):

`particle_selection_generation` (JA, language_id=3):

```
対象語: {lemma}（{pos}）  語義: {definition}
文: {p1_sentence}

上の文から助詞を1つ選び、その助詞を「＿＿」に置き換えてください。選ぶ助詞は、
学習者が混同しやすいもの（は/が、に/で、を/が など）を優先してください。

出力（JSONのみ、数字キー）:
{"1": "<助詞を＿＿に置換した文>",
 "2": "<正解の助詞>",
 "3": [ {"1":"<誤答助詞>","2":"<なぜこの文では誤りか（日本語）>","3":"<error_tag>"} ×3 ],
 "4": "<正解の理由（日本語）>"}

制約: 誤答助詞はすべて実在の助詞で、この文では文法的または意味的に不適切であること。
2つ以上の助詞が自然に成立する文は選ばないこと。
```

`particle_judge` (JA): given blanked sentence + key + distractors, rate each distractor 1–5 on "could this particle also yield a natural sentence here?" (5 = clearly wrong → keep; ≤2 → reject). Mirrors the collocation-judge contract so `judges/` code is reused.

**JA P1 additions** (vs the ZH P1 it is cloned from): output key for `register` (敬語 level: plain/polite/honorific/humble/formal/casual), readings in kana per sentence occurrence, and counter word (助数詞) for concrete nouns — the JA analogue of ZH rule 18's measure word.

`kanji_to_reading`/`hanzi_to_pinyin`/`tone_id_word` need **no prompts** — readings from UniDic/pypinyin; confusion sets algorithmic.

### 6.7 Traditional Chinese support (operator answers, final)

**Mechanism (decided): dual-store both scripts at generation time.** Simplified remains canonical for keys, matching, and identity; a traditional mirror is persisted alongside so serving does zero conversion.

- `opencc` dependency (config `s2twp`, phrase-level idiom handling) is used **once, at generation/enrichment time**, never at serve time.
- New column `dim_vocabulary.lemma_traditional text`, filled at enrichment (P0.9). Ambiguous one-to-many lemmas (发→發/髮 class) resolved with jieba word context; the stored value is authoritative.
- Every ZH exercise's `content` carries a **persisted traditional mirror**: `content.hant = { "prompt": {"stem": …}, "options": [{"text": …}, …], "reasoning": […] }` — generated by the renderer immediately after validation, covering every learner-visible TL string. QA exceptions are corrected in a `script_conversion_overrides(simplified PRIMARY KEY, traditional, note)` table consulted at generation; fixing an override re-renders only the `hant` mirror (idempotent backfill script).
- Serving picks `content.hant` fields when `users.exercise_preferences->>'script_variant' = 'traditional'`, falling back to simplified per-field if a mirror is missing (and flagging it). Pinyin, tone, audio, and keys are script-independent; typed ZH answers are normalised `t2s` before matching (`cloze_typed`).
- Cost: ~2× the ZH text payload inside `content` (storage is trivial at this scale) + one mirror-render step per item; benefit: no conversion logic or risk in the serving path.
- **Scope (decided): practice/vocab surfaces first.** Tests, mysteries, and UI chrome are a separate later epic reusing the same dual-store pattern.

### 6.8 Difficulty calibration

Three existing layers + one addition: (1) authoring difficulty via age-tier instruction pinned to `definition_level`; (2) nightly IRT 2PL over attempts (defaults a=1.0, b=0.0 until n≥20); (3) serving adaptivity via the unified score's IRT-information term. **Addition:** seed `exercises.irt_difficulty` at generation — `b₀ = z(frequency_rank percentile) × 0.5 + level_offset(ladder_level)` — so cold items rank sensibly; calibration overwrites as data arrives.

### 6.9 Language branching (single decision point)

All language variance resolves at **routing time** from `dim_exercise_capabilities` + the language validation profile; generators stay language-blind (prompts, tokenizers, confusion-set providers injected):

```python
def plan_generation(sense) -> GenerationPlan:
    caps = capabilities[sense.language_id]             # dim_exercise_capabilities
    profile = get_validation_profile(sense.language_id)
    pos_class = sense.semantic_class                   # ratified 6-value enum

    plan = []
    for cap in caps:
        if not cap.is_enabled or pos_class not in cap.pos_classes: continue
        missing = [r for r in cap.requires if not satisfied(sense, r)]
        if missing:
            if cap.generator == 'deterministic' and recoverable(missing):
                enrich(sense, missing)                 # pypinyin / UniDic / classify / opencc — idempotent
            else:
                record_skip(sense, cap.type_code, missing)
                continue
        plan.append(cap)
    return plan
# Providers per language:
#   zh: jieba + pinyin_service(sandhi) + classifier_dict + homophone_index + opencc
#   ja: fugashi/UniDic + kana_confusion_sets + counter_dict (P3) + kanji components (P3)
#   en: spaCy chunker
```

### 6.10 Fallback logic (per missing datum)

| Missing | Behaviour |
|---|---|
| `semantic_class` | LLM-classify inline (cached); on failure → treat as `abstract` + flag for review |
| `pronunciation` (ZH/JA) | Derive deterministically; unparseable lemmas skip reading/tone types with warning |
| `morphological_forms` < 2 | Skip L4/word_family with warning (shipped, language-aware) |
| ZH concrete noun not in classifier dict | L4 omitted (`active_levels` drops 4); never block the sense |
| JA concrete noun not in counter dict | Fall back to `particle_selection` for L4 |
| `primary_collocate` absent/judge-rejected | Skip L5/L8; drop collocation from required families (existing mechanics) |
| `register` (JA) absent | L6 register-error sentences disabled for that sense (domain/restriction errors only) |
| P1 < 6 acceptable sentences after repair | **Block the sense** → `generation_queue(reason='regen')` |
| TTS failure | Ship text variant; queue audio backfill (existing tab) |
| `content.hant` mirror missing/partial | Serve simplified for the missing fields + flag to `script_conversion_overrides` review / mirror backfill |
| Judge template/model missing | **Fail closed in batch** (block + alert) |

Never destructive: regeneration renders to a staging list and only replaces on non-empty success (audit B1, ships in P1.3).

### 6.11 Scheduling integration (no new scheduler)

Four contact points with the existing stack: (1) corrected `dim_exercise_types.family` rows (P0.3); (2) realistic `expected_seconds` per type (45 s flat is wrong for speed-round/jumbled; nightly P50 refresh already specced); (3) per-family inventory coverage (§6.3); (4) `ladder_level` + `tags.variant` populated so variant alternation works. New types slot into rings via `ladder_level` in the capability matrix (`particle_selection`/`counter_match`/`classifier_match`→L4; readings/tone→Ring 1).
